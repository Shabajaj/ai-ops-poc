# Incident 002: optionchain service — build failures after JDK base image bump

**Severity:** Sev3
**Service:** optionchain
**Status:** Resolved

## Summary
A routine base-image update bumped the JDK version used to build the
`optionchain` service. The build started failing with compilation errors on
classes that use Lombok annotations (`@Getter`, `@Setter`, `@Builder`, etc.)
— fields and methods that Lombok normally generates at compile time were
reported as missing, even though the source code hadn't changed.

## Symptoms
- CI build failures immediately after merging the base image bump, with no
  application code changes in the same window
- Compiler errors like "cannot find symbol" pointing at getter/setter methods
  that only exist via Lombok-generated bytecode, not in the visible source
- Locally reproducible only on machines that had also picked up the newer
  JDK — engineers still on the older JDK locally could not reproduce it,
  which delayed triage
- Not a runtime crash — this was caught at build time before anything ran in
  a cluster, so no CrashLoopBackOff or pod-level symptoms

## Root Cause
Lombok works by hooking into the Java compiler's annotation processing at
build time to generate boilerplate code. Newer JDK versions changed internal
compiler APIs that Lombok's annotation processor depended on. The specific
Lombok version pinned in the project's build file predated JDK compatibility
fixes for the new compiler internals, so annotation processing silently
failed to generate the expected members — the compiler then correctly (from
its point of view) reported those members as missing.

This is a version-compatibility gap between three things that normally don't
get reasoned about together: the base image's JDK version, the pinned Lombok
version, and the build tool's annotation processing configuration. Bumping
just the base image, without checking Lombok's compatibility matrix for that
JDK, was enough to break the build even though no application code changed.

## Fix
Pinned Lombok to a version confirmed compatible with the new JDK (per
Lombok's published compatibility notes), and added an explicit
`-parameters` / annotation-processing flag check to the build to make the
dependency between JDK version and Lombok version visible in one place
instead of being implicit.

## Detection Gap
The base image bump was treated as a routine, low-risk dependency update and
merged without a compatibility check against annotation-processing tooling.
There was no automated check cross-referencing "JDK version in base image"
against "Lombok version in build file" before this incident.

## Prevention
- Added a CI check that fails the build (with a clear message) if the JDK
  major version and pinned Lombok version fall outside Lombok's documented
  compatibility matrix, instead of letting it fail with a confusing
  "cannot find symbol" error
- Base image bumps that touch the JDK version now require running the full
  build (not just a smoke test) against a representative service using
  Lombok before merging, specifically because this failure mode is silent
  until build time and gives no runtime signal at all
